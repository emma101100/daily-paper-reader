## 摘要
基于大语言模型的智能体能够通过结合推理与环境反馈来解决多步交互任务，但每个片段都从相同的固定上下文开始，任务结束后途中发现的任何有用策略也会随之丢失。现有方法要么将学习限制在当前任务内，要么将所有经验汇集到一个无类型区分的单一存储中，不区分知识类型，不通过使用来跟踪质量，也不平衡知识库仍然欠缺的内容。我们提出统一上下文演化（UCE），这是一种无梯度的框架，它将智能体经验外部化为一个不断演化的类型化可演化上下文单元（ECU）库。UCE将经验分解为四种互补类型（记忆、策略、工作流和技能），每种类型都根据类型特定条件从轨迹中生成，在决策时检索，通过重复使用结果进行评分，并在不再有价值时进行剪枝。调度模块将每个周期的生成预算分配给知识库中最为薄弱的类型。在两个交互式基准测试中，UCE将ALFWorld成功率从75.4%提升至96.3%，将WebShop任务得分从45.1%提升至61.3%，并且积累的知识库无需重新训练即可迁移到替代的智能体骨干网络上。

## Abstract
LLM-based agents can solve multi-step interactive tasks by combining reasoning with environment feedback, yet each episode starts from the same fixed context and any useful strategy discovered along the way is lost once the task ends. Existing approaches either limit learning to the current task or pool all experience into a single untyped store, without distinguishing knowledge types, tracking quality through use, or balancing what the library still lacks. We introduce Unified Context Evolution (UCE), a gradient-free framework that externalizes agent experience into an evolving library of typed Evolvable Context Units (ECUs). UCE decomposes experience into four complementary types (Memory, Strategy, Workflow, and Skill), each generated from trajectories under type-specific conditions, retrieved at decision time, scored through repeated usage outcomes, and pruned when no longer valuable. A scheduling module allocates each cycle's generation budget toward the types where the library is weakest. Across two interactive benchmarks, UCE raises ALFWorld success from 75.4% to 96.3% and WebShop task score from 45.1% to 61.3%, and the accumulated library transfers to alternative actor backbones without retraining.

### 1. 论文的核心问题与整体含义

*   **核心问题**：基于大语言模型的交互式智能体在执行多步任务时，面临严重的经验流失问题。每个新任务开始时，智能体都从相同的固定系统指令和少样本示例出发，上一个任务中发现的任何有效策略或环境机制都会在任务结束时消失，导致智能体无法从过往交互中持续学习和进步。
*   **研究动机**：
    *   现有方法要么将学习限制在单个任务内（如自我反思），改进无法跨任务复用。
    *   要么将所有经验混合存储于一个无类型区分的知识库中，无法有效区分、追踪和管理不同性质的知识（如“如何做”的程序性知识与“是什么”的事实性知识）。
    *   缺乏一种机制来评估知识库中各类型知识的质量，并根据知识库的能力短板来动态引导新知识的生成，实现平衡演化。
*   **整体含义**：论文旨在提出一个框架，让智能体的“上下文”本身成为学习的载体。通过将经验外化为一个结构化、可演进的类型化知识库，使智能体能够在不更新模型参数（无梯度）的情况下，跨任务持续积累、评估和调用不同类型的知识，从而显著提升任务性能。

### 2. 论文提出的方法论

*   **核心思想**：统一上下文演化（UCE）框架，其核心是维护一个不断演化的**可演化上下文单元（ECU）库**。
*   **关键技术细节**：
    *   **经验分解（四种ECU类型）**：UCE将智能体的经验分解为四种功能互补、自然语言描述的知识类型，每种类型有独立的定义、生成条件和检索范围：
        1.  **记忆**：描述环境机制如何运作。从单条轨迹中提取。例如：“水槽可以清洁便携物品，无需开关水龙头。”
        2.  **策略**：在特定条件下智能体应如何决策。从同一任务类型的多条轨迹中提取。例如：“如果放置失败，打开容器并重试。”
        3.  **工作流**：智能体应遵循的执行序列。从同一任务类型的多条成功轨迹中归纳。例如：“找到目标，处理它，然后将其移动到目标容器。”
        4.  **技能**：可复用的子操作方法。从跨任务类型的轨迹中提取。例如：“搜索容器的系统方法：打开关闭的容器，找到后立即拾取目标。”
    *   **ECU全生命周期管理**：
        1.  **表示**：每个ECU是一个五元组（类型、任务类型、检索条件、内容、适应度得分）。
        2.  **检索**：在决策时，根据语义相似度、任务类型匹配度和适应度得分综合打分，为每类ECU独立检索Top-K条目注入到提示词中。
        3.  **适应度得分**：使用拉普拉斯平滑的基于使用的得分 `fe = (se + 1) / (ue + 2)`，其中`se`是累计成功次数，`ue`是累计使用次数。新生成的ECU默认得分为0.5，其价值通过实际使用来验证。
        4.  **剪枝与去重**：移除得分低于阈值或与同组其他单元语义高度相似且得分落后的ECU。
    *   **知识产出调度（KYS）**：一个用于分配每轮演化中生成预算的模块，以弥补知识库的短板。它通过三层机制决定每种ECU类型的生成数量：
        *   **可行性过滤**：检查某类ECU是否还有生成空间（如是否还有任务类型缺乏足够的高质量ECU）。
        *   **需求评估**：量化当前的知识缺口，结合覆盖率和质量差距来计算需求度。
        *   **产出追踪**：维护每种类型ECU对性能提升贡献的历史指数移动平均（EMA），用于评估生成的有效性。
        最终通过加权融合需求度和历史产出来计算每种类型的综合得分，并按比例分配生成预算。
    *   **演化周期**：UCE通过五个循环阶段运行：评估 -> 收集 -> 调度 -> 生成 -> 清理。评估集和收集集不相交，以保证评估的公正性。适应度得分仅在收集集中更新。

### 6. 论文的主要结论与发现

*   **性能显著提升**：UCE在两个基准上都取得了稳定且显著的效果提升。在ALFWorld上，成功率从ReAct基线的75.4%提升至96.3%；在WebShop上，任务得分从45.1%提升至61.3%，购买成功率从30%提升至42%。
*   **知识可迁移**：积累的ECU库可以被直接注入到不同的智能体推理框架中，无需任何重训练即可提升其性能，证明了知识的通用性和框架的解耦性。

*   **知识产出调度有效**：KYS机制能够有效识别知识库的短板，并将生成预算分配给最需要的知识类型，实现了知识库的平衡演化。

### 7. 优点

*   **创新性的知识解耦**：将智能体经验创造性地分解为四种功能明确、生命周期独立的ECU类型，相较于无类型知识库，管理更精细，利用效率更高。
*   **全生命周期质量管理**：引入基于使用的动态适应度得分来追踪ECU的实际价值，并包含主动的清理和去重机制，确保知识库的高质量与精简性。
*   **自适应的演化调度**：KYS模块能像“项目经理”一样自动评估知识库短板并引导学习方向，使演化过程更高效、更具方向性。
*   **无梯度且模型无关**：该方法不需要对LLM进行梯度更新或微调，且生成的ECU库独立于特定的智能体提示词格式，可直接迁移至不同骨干网络，具有极佳的兼容性和低应用门槛。

### 8. 不足与局限

*   **环境与模型相对单一**：实验仅在两个相对小规模的文本交互环境中进行，并使用了一对特定的闭源模型。结果在更大规模、更开放、涉及连续动作或多模态输入的环境（如VisualWebArena）中的泛化能力有待验证。同时，表现提升多大程度上依赖于GPT-5.2的强大生成能力尚不明确。
*   **任务规模限制**：实验任务集较小，KYS调度信号的稳定性和知识库在大规模任务集上的持续演化能力（是否会出现饱和或退化）仍是未解决的问题。
*   **ECU类型预制**：四种ECU类型是人工设计的。论文指出，一个能够自动发现和定义新知识类型的机制，将是未来使该框架适应新领域的重要方向。
*   **知识边界**：错误分析表明，UCE善于提供“做什么”的规则和“怎么做”的程序，但在需要从界面选项中精确操作（“具体点哪个”）的场景下提升有限，这可能是基于提示词的知识注入方法本身的局限。
