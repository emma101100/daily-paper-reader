## 摘要
多智能体大语言模型系统——编码智能体、运维智能体、文档智能体——如今经常针对同一个代码库、Kubernetes 集群或文档并行运行多个智能体。只要其中两个对共享状态进行修改，就会进入经典并发控制研究了几十年的领域，但经典机制并不适合大语言模型智能体。单个智能体事务持续数分钟的推理时间，读集广泛且不透明而无法静态推断，并且智能体所作用的实时状态既不能分叉也不能缓冲，因此写操作在执行时立即生效。锁会阻塞长时间的推理间隔；乐观并发控制的中止并重试会在每次冲突时丢弃数分钟的工作。本文构建的并发控制利用了经典事务所缺乏的一种能力：每个智能体内部的大语言模型能够判断冲突写入是否使其计划失效，并能够精确修复依赖该写入的操作。因此控制变为建议性的：运行时通知，智能体修复。我们的协议 MTPO（单调轨迹预序）在启动时确定一个串行化顺序，为每个读取提供按该顺序过滤的值，并投机性地就地应用写入；单向通知要求受影响的读取者重新判断并修补其计划，同时框架通过每个工具预先注册的 saga 风格逆操作，机械地撤销和重排错位的写入。在静止状态下，运行结果在预先决定的顺序下是可串行化的。我们将 MTPO 实现为 CoAgent，这是一个工具调用中间件，其特权工具工厂在线生成具有足迹声明且可撤销的工具。在十个有竞争的工作负载上，CoAgent 在 95% 的串行正确性范围内，实现了 1.4 倍的加速和接近串行的令牌消耗，而两阶段锁和乐观并发控制几乎放弃了所有并发增益；在仅使用 bash 的目标系统上，它在线构建了包含 25 个工具的库，并将任务通过率从 45/71 提升至 63/71，时间开销为 0.80 倍，成本为 0.86 倍。

## Abstract
Multi-agent LLM systems -- coding agents, devops agents, document agents -- now routinely run several agents in parallel against the same git tree, Kubernetes cluster, or document. As soon as two of them mutate shared state, they enter the regime classical concurrency control has studied for decades, but classical mechanisms fit LLM agents poorly. A single agent transaction spans minutes of inference, read sets are broad and opaque rather than statically inferable, and the live state agents act on admits neither fork nor buffer, so writes take effect the moment they execute. Locks block long inference intervals; OCC abort-and-retry discards minutes of work on every conflict.   This paper builds concurrency control on a capability classical transactions lack: the LLM inside each agent can judge whether a conflicting write invalidates its plan, and can repair exactly the operations that depended on it. Control therefore turns advisory: the runtime informs, the agent repairs. Our protocol, MTPO (Monotonic Trajectory Pre-Order), fixes a serialization order at launch, serves each read the order-filtered value, and applies writes speculatively in place; a one-way notification asks an affected reader to re-judge and patch its plan, while the framework mechanically undoes and reorders misplaced writes through the saga-style inverse each tool registers in advance. At quiescence the run is serializable in the pre-decided order. We realize MTPO as CoAgent, toolcall middleware whose privileged ToolSmith grows footprint-declared, undoable tools online. On ten contended workloads, CoAgent stays within 5\% of serial correctness at a $1.4\times$ speedup and near-serial token cost, where 2PL and OCC surrender nearly all concurrency gains; on a bash-only target system, it grows a 25-tool library online and lifts the task pass rate from 45/71 to 63/71 at $0.80\times$ the time and $0.86\times$ the cost.

### 1. 核心问题与研究动机（背景）
- **问题**：在 LLM 驱动的多智能体系统中，多个代理并行操作同一个共享状态（如代码库、K8s 集群、文档）时，读写的交错极易导致并发异常（如读取过期数据、丢失更新），且这类故障已在真实生产规模的跟踪中被证实。传统的并发控制机制（两阶段锁 2PL 与乐观并发控制 OCC）直接迁移到智能体场景存在严重缺陷：
  - **性能缺口**：LLM 推理事务持续数分钟甚至数小时，2PL 长时间持有锁导致大量阻塞和死锁；OCC 在冲突时需要整事务重试，抛弃大量已完成的工作和推理代价。
  - **功能缺口**：真实外部状态（如 K8s 集群、生产数据库）无法被“分叉与合并”，OCC 无法将修改缓冲在私有工作区，2PL 回滚也难以撤销已生效的外部副作用。
- **整体含义**：亟需一种能适应长推理、立即生效写入、读集广泛且不可预知的智能体并发控制机制，既能保证可串行化正确性，又不过度牺牲并发度与推理成本。

### 2. 方法论：核心思想与关键技术细节
- **核心思想**：利用 LLM 的语义理解能力，将强制性的并发控制（阻塞/中止）改为**建议性**（advisory）机制：运行时仅通知代理可能发生冲突，由代理自己判断冲突是否真正影响其计划，并精确修复受影响的局部操作；错位的写入由框架利用预注册的逆操作机械地顺序化，无需代理参与。
- **协议：MTPO（单调轨迹预序）**
  - **固定串行顺序**：启动时为所有代理分配一个全局的串行化顺序（$\sigma$ 顺序），所有冲突均按此顺序定向。
  - **过滤读**：每次读取返回的值是对象写轨迹中所有 $\sigma$ 排名在读者之前的写入的复合结果（FilterRead），保证所见即串行视图。
  - **投机写入**：写操作立即就地执行，不论其物理执行时间和串行顺序是否一致。若写操作到达时其 $\sigma$ 位置早于已应用的某些写，则视为“迟到写”。对于盲写，仅记录轨迹，不覆盖当前状态；对于读改写，框架会撤销对象上高于当前排名的写，应用本次写入，再重新应用被撤销写（需要工具的逆操作）。
  - **单向通知**：当低序号写者修改了某个对象，而高序号读者已经读过该对象的旧值时，框架向高序号读者推送一条通知（携带更新后的值），读者在下一次推理时判断该变化是否令原计划失效，然后仅修补依赖此前提的操作（或忽略无关变更），避免整体重试。
  - **逆操作与工具工厂（ToolSmith）**：每个工具定义时附带 `reverse` 片段，用于机械撤销。对于不可逆操作（如发送邮件），框架会阻塞直至所有低序号代理都已提交。为降低声明负担，引入特权代理 ToolSmith，用探索阶段自动生成对象树和带足迹声明的工具，Worker 通过自然语言请求新工具，ToolSmith 在线合成并向工具库追加。

- **技术架构**：CoAgent 中间件，截获所有工具调用，维护对象树和写轨迹，实现过滤读、逆操作、通知推送，保证静止状态下最终状态的串行化等价。

### 3. 实验设计：数据集 / 场景、基准与对比方法
- **数据集与场景**：
  - **WorkBench**：模拟办公自动化（CRM、日历、邮件、分析、项目管理），选取 5 个独立任务作为 Agent‑1，手工构造一个与之产生典型并发冲突的 Agent‑2，形成 5 个竞争单元（cells）。
  - **AIOpsLab**：基于真实 K8s 集群的微服务事件调查与修复任务，同样方法构造 5 个单元，共 **10 个竞争工作负载**。
  - **AIOpsLab 全量任务（71 个）**：用于测试在仅有 `bash` 的原生环境下，CoAgent 的工具库在线增长及最终效果。
- **对比方法**：
  - `serial`：顺序执行两个代理（正确性与成本上界）。
  - `naive`：无任何并发控制，并行运行（速度下界，错误率高）。
  - `2PL + saga`：两阶段锁，死锁时用 saga 逆操作回滚受害者并重启。
  - `OCC + saga`：乐观并发控制，提交时验证失败则中止并整事务重试。
  - `MTPO`（本文方案）。
- **评价指标**：串行正确性（满足不变量的试验比例）、壁钟加速比（相对 serial）、令牌消耗比（相对 serial）。

### 4. 资源与算力
- **硬件**：所有试验运行在一台双路 Intel Xeon E5‑2650 v4 服务器上（24 物理核，125 GiB 内存，1.8 TB NVMe SSD，无 GPU）。K8s 场景使用同主机上的单节点 kind 集群。
- **模型调用**：Worker 代理使用 `deepseek‑v4‑flash`，通过 OpenRouter API 访问；ToolSmith 及全量 71 任务实验使用 `deepseek‑v4‑pro`。文中未提及 GPU 型号或本地推理，主要推理成本为 API 调用令牌费用。
- **算力需求未显式给出**，但所有测量均基于 API，实验主要衡量时间与令牌开销，非传统 GPU 训练任务。

### 5. 实验数量与充分性
- **竞争负载实验**：对 10 个单元（5 WorkBench + 5 AIOpsLab）× 5 种协议，每组合重复 **10 次**，共计 500 次试验。
- **案例研究**：选取最具代表性的“金丝雀异常”单元，用时间线详细对比四种协议（naive / 2PL / OCC / MTPO）一次运行中的内部行为。
- **工具库在线增长实验**：对 71 个 AIOpsLab 任务顺序执行，记录工具库累积曲线、Worker 与 ToolSmith 的最终任务通过数、总时间及总令牌成本，与纯 `bash` 代理对比。
- **充分性与公平性**：覆盖了两种不同领域的基准测试，手工构造的冲突单元均带有预设不变量，确保正确性可客观验证；采用了相同的模型和 API 环境，对比协议均在同一中间件层实现，排除了系统差异。整体实验设计较为严谨，但未测试大规模（>10 个代理）或更复杂的嵌套冲突场景。

### 6. 主要结论与发现
- **正确性与效率兼得**：在 10 个竞争工作负载上，MTPO 正确性保持在 **95%** 左右（5 个试验出现接收者误判通知无关），壁钟加速比达 **1.4×**，令牌成本仅为串行的 **1.15×**。而 2PL 死锁频率 0.81/次，加速比退化至 1.04×；OCC 中止率 0.95/次，速度反而慢于串行（0.93×），成本高达 1.83×。
- **选择性修复显著降低开销**：案例显示，MTPO 仅在冲突写发生后 6.4 秒内完成局部修补，避免了 OCC 的 29 秒全量重做。
- **ToolSmith 可在线构建可撤销工具库**：在 bash‑only 环境下，CoAgent 从零开始经 71 个任务自动积累 25 个工具（含 11 个快照读、2 个实时读、12 个可逆写），并使任务通过率从 45/71 提升至 **63/71**，时间 0.80×，成本 0.86×。

### 7. 优点（亮点）
- **利用 LLM 能力解决 CC 难题**：将并发控制从强制性改为建议性，让语言模型完成传统事务无法企及的语义判定和局部修复，是新范式。
- **形式化保证**：给出了单调轨迹预序协议及其可串行化证明（在静止状态下），理论基础扎实。
- **在真实外部状态上可行**：通过逆操作和写入轨迹管理，解决了“不能分叉/不能回滚”的行动成本函数问题，可直接用于 Kubernetes 等外部系统。
- **在线工具扩展**：ToolSmith 机制减轻了声明每个对象和逆操作的人工负担，使方案能自动适应新环境，具有实际部署的可行性。
- **实验对比全面**：涵盖典型并发控制方案，同时展现正确性、加速比、成本的三维权衡，说服力较强。

### 8. 不足与局限
- **代理自我修复能力假设强**：要求代理能准确判断通知的相关性，实验中仍有 ~5% 的误判（A3 假设不完美）。若模型能力不足，可能引入静默的不一致。
- **不可逆操作失去并发性**：对于无法提供逆操作的工具（如支付、外部邮件），必须等所有低序代理提交后才能执行，这部分写入退化到**接近串行**。
- **实验规模受限**：仅测试了 2 代理并行的场景，未验证大规模多代理（如 Kimi 的百级代理）下的性能与正确性；10 个竞争单元虽针对性强，但多样性仍显不足。
- **系统环境单一**：所有实验在同一台服务器和 K8s kind 集群上进行，未涉及分布式、跨云或异构系统的复杂情况。
- **对模型依赖的重成本**：通知之后模型的每一次推理都可能增加额外令牌，如果模型能力波动，可能导致意外修复失败或成本上升；对低成本弱模型的适用性未探讨。
