# ABSTRACT 

Large language model (LLM)-based agents increasingly solve complex tasks by interacting with external tools, retrieval systems, memory modules, environments, and other agents. While these capabilities expand agent autonomy, they also make agent behavior harder to verify, debug, and audit. Final-answer accuracy alone cannot explain how an output was produced, which evidence supported each claim, whether tool calls were justified, how memory influenced later decisions, or where execution failures originated. Evidence tracing and execution provenance address this gap by modeling how retrieved evidence, tool outputs, memory items, environment observations, intermediate claims, actions, and final answers are connected throughout agent execution. This survey provides a systematic review and conceptual framework for evidence tracing and execution provenance in LLM agents. We organize related work around a unified provenance perspective that connects retrieval grounding, claim support, tool-use safety, memory lineage, observability, debugging, audit, and recovery. We introduce a taxonomy covering trace sources, evidence and execution units, provenance relations, tracing granularity, tracing timing, representation forms, and trust functions. We then review key methodological directions, including provenance representation, evidence attribution, tool-use provenance, runtime guardrails, provenance-bearing memory, trace-based observability, and failure diagnosis. We further map existing benchmarks, datasets, and evaluation metrics to provenance-related capabilities, and discuss how evaluation can move from final-answer correctness toward process-level accountability. Finally, we outline open challenges, including unified trace schemas, claim-level and semantic provenance, provenance-aware safety mechanisms, realistic execution-trace benchmarks, recovery-oriented evaluation, and privacy-aware audit infrastructure for trustworthy LLM agents. 

Keywords LLM Agents, Agent Traces, Evidence Tracing, Execution Provenance, Provenance Graphs, Tool-Use Safety, Memory Provenance, Agent Observability, Runtime Guardrails, Trustworthy AI

We use evidence tracing to refer to mechanisms that identify, record, and connect the evidence units that support, contradict, invalidate, or influence agent claims, decisions, and actions. Under this view, agents perform reasoning and action, while provenance structures serve as an accountability layer that records how evidence, tools, memory, observations, claims, actions, and agent interactions jointly shape an execution outcome. Evidence tracing focuses on support and influence relationships between evidence and agent behavior; execution provenance captures the wider causal, procedural, and dependency relationships among evidence, computation, action, and outcome. # 2 Background: LLM Agents, Traces, and Trust 

This section introduces the background concepts needed for studying evidence tracing and execution provenance in LLM agents. We first describe LLM agents and agentic workflows, then summarize the main forms of traces produced during agent execution, and finally explain why traceability and provenance are necessary for trustworthy agent behavior. 

2.1 LLM Agents and Agentic Workflows 

Large language model (LLM)-based agents extend standalone language models from passive response generation to goal-directed task execution. Instead of producing a single response from a static prompt, an LLM agent can interpret task context, reason over intermediate states, invoke external tools, retrieve information, consult memory, interact with environments, and coordinate with other agents. This execution-oriented view distinguishes LLM agents from conventional language generation systems, where the final textual output is often the primary object of evaluation. Representative agent paradigms illustrate this shift. ReAct interleaves reasoning traces with task-specific actions, showing that intermediate reasoning and external observations can jointly guide agent behavior Yao et al. [2023a]. Reflexion introduces verbal feedback and episodic memory, allowing agents to revise future behavior based on previous failures Shinn et al. [2023]. Generative Agents further demonstrate how observation, planning, reflection, and memory can support long-term interactive behavior in simulated environments Park et al. [2023]. Tool-oriented systems such as Toolformer and ToolLLM extend LLMs with external tool and API access, making tool calls, arguments, and execution results central components of agent workflows Schick et al. [2023], Qin et al. [2024]. Benchmarks such as AgentBench, WebArena, and τ -bench reflect this broader view by evaluating LLMs in multi-step, interactive, tool-using, and environment-grounded settings Liu et al. [2024], Zhou et al. [2024], Yao et al. [2025]. Multi-agent frameworks such as AutoGen and CAMEL further expand the execution space by introducing role assignment, inter-agent communication, delegation, and collaborative task solving Wu et al. [2024a], Li et al. [2023]. Under this view, an LLM agent is not merely a model that generates an answer; it is a system that produces an execution trajectory. This trajectory may include internal reasoning, external evidence, tool calls, environment observations, memory operations, and interactions with other agents. Understanding such trajectories is essential for studying evidence tracing and execution provenance. 

2.2 Execution Traces in LLM Agents 

Because LLM agents operate through iterative interaction, their behavior naturally produces execution traces. In this survey, an agent trace refers to the recorded sequence of information items, reasoning states, actions, observations, and outputs generated or consumed during an agent run. The concrete structure of a trace depends on the agent architecture and task environment, but common trace elements include user instructions, reasoning steps, plans, reflections, retrieval queries, retrieved documents, tool calls, tool parameters, tool outputs, memory reads and writes, environment observations, intermediate claims, inter-agent messages, actions, and final answers. Different agent paradigms expose different types of trace elements. Reasoning-action agents emphasize reasoning traces, selected actions, and external observations Yao et al. [2023a]. 2.3 From Final-Answer Accuracy to Process-Level Accountability 

Traditional evaluation often focuses on whether the final answer is correct. While useful, final-answer accuracy is insufficient for understanding LLM agents because agent outputs are produced through multi-step execution. Before producing an answer, an agent may retrieve evidence, call tools, modify external states, update memory, observe an environment, or exchange messages with other agents. As a result, two executions with the same final answer may 6From Agent Traces to Trust differ substantially in reliability, safety, and auditability. This limitation becomes especially important when agent actions have external consequences: a correct answer may still be produced through an unnecessary, unsafe, or policy-violating tool call, while an incorrect answer may originate from misleading evidence, erroneous tool outputs, stale memory, malformed parameters, invalid intermediate claims, environment misobservation, or failures in multi-agent coordination Yao et al. [2023a], Qin et al. [2024], Zhou et al. [2024], Yao et al. [2025]. This issue is already visible in retrieval-augmented generation and evidence-based generation. Work on citation generation, factual precision, RAG evaluation, hallucination detection, and citation quality shows that answer correctness should be separated from evidence support and attribution quality Gao et al. [2023], Min et al. [2023], Es et al. [2024], Saad-Falcon et al. [2024], Ru et al. [2024], Niu et al. [2024], Wu et al. [2024b], Schreieder et al. [2025]. For example, a generated answer may be factually correct but cite an irrelevant source, or it may cite a relevant document while making a claim that the document does not support. These issues become more complex in agent settings, where the support for an output may come not only from retrieved documents but also from tool outputs, memory records, environment states, and prior agent messages. Agent benchmarks further reinforce the need for process-level accountability. In web, tool-use, and user-interaction settings, success depends not only on producing a plausible response but also on following policies, invoking tools correctly, updating external states safely, and recovering from intermediate errors Zhou et al. [2024], Yao et al. [2025]. Recent work on LLM-agent hallucination similarly argues that agent failures should be analyzed across the complete workflow, since hallucinations may arise during planning, tool use, memory access, observation interpretation, or final response generation Lin et al. [2025]. Studies of multi-agent failure also show that system-level failures can result from specification errors, inter-agent misalignment, and inadequate task verification, rather than isolated answer-level mistakes Cemri et al. [2026]. Therefore, evaluating agents requires asking not only whether the final result is correct, but also how the agent reached it: what it observed, what evidence it used, which tools it called, which memory items it relied on, what actions it took, and how intermediate states contributed to the final outcome. This process-level view is also central for debugging. Without traces, failures from task interpretation, retrieval, tool selection, parameter generation, tool execution, memory retrieval, state tracking, inter-agent communication, and final response synthesis collapse into a single endpoint failure. With traces, failures can be attributed to specific steps, enabling targeted repair, safer deployment, and more meaningful evaluation. 

2.4 Traceability, Provenance, and Trustworthiness 

Traceability and provenance provide the conceptual basis for process-level accountability in LLM agents. In this survey, 

traceability refers to the ability to record and inspect the execution path of an agent, including its reasoning steps, actions, observations, tool calls, memory accesses, inter-agent messages, and outputs. Provenance goes further by modeling the relationships among these trace elements, such as which evidence supports a claim, which observation triggered an action, which tool output updated a memory item, or which intermediate decision contributed to the final answer. This view builds on broader provenance and tracing traditions: W3C PROV-DM models provenance through entities, activities, and agents involved in producing an object, while OpenTelemetry represents distributed executions as traces composed of spans and causal relationships World Wide Web Consortium [2013], OpenTelemetry [2026]. This distinction matters because raw logs alone are not sufficient for trustworthy LLM agents. A chronological log may show that an agent retrieved a document, called a tool, and produced an answer, but it may not explain whether the document actually supported the answer, whether the tool call was necessary, whether the tool output contradicted retrieved evidence, or whether a memory item was stale. LLM agent provenance therefore requires more than operational logging. It must capture semantic relationships among heterogeneous execution artifacts: a retrieved document may support a claim, a tool output may contradict a memory item, an observation may invalidate a previous assumption, or an untrusted instruction may influence a later tool call. Provenance structures transform traces from passive records into accountability artifacts that can support verification, attribution, debugging, auditing, recovery, safety enforcement, and trust calibration. The need for provenance is especially strong in safety-critical and security-sensitive agent systems. Tool-integrated agents can be exposed to indirect prompt injections, untrusted web content, unsafe tool outputs, or malicious instruc-tions Ruan et al. [2024], Zhan et al. [2024], Debenedetti et al. [2024]. Recent work on prompt-injection defenses, information-flow control, taint tracking, runtime enforcement, and execution bounding shows that trustworthy agent execution depends on tracking how information flows through the agent and how it influences actions Debenedetti et al. [2025], Costa et al. [2025], Cai et al. [2026], Wang et al. [2025a], Sequeira et al. [2026], Bühler et al. [2025]. These methods move beyond final-output checking toward runtime control over the provenance of information, instructions, and actions. 7From Agent Traces to Trust Agent observability and debugging systems point in the same direction. Observability frameworks record execution traces for inspection, while debugging systems attempt to localize failures from agent logs and traces Dong et al. [2024], AlSayyad et al. [2026], Deshpande et al. [2025], Rorseth et al. [2025]. Multi-agent failure studies further show that failures may arise from specification mismatches, inter-agent misalignment, weak verification, or cascading errors across interaction steps Cemri et al. [2026]. These findings suggest that agent trustworthiness cannot be reduced to model accuracy alone; it depends on whether the execution process can be reconstructed, inspected, and governed. In this survey, we therefore treat evidence tracing and execution provenance as the accountability layer of LLM agent systems. Agents perform reasoning and action, while provenance structures record, connect, and evaluate the evidence, observations, tool outputs, memory items, intermediate claims, and actions that contribute to final behavior. This background motivates the taxonomy developed in the following sections, where we examine what should be traced, how trace elements can be represented, how provenance relations can be modeled, and how provenance can support evaluation, diagnosis, and safe agent deployment. 

# 3 Taxonomy of Evidence Tracing and Execution Provenance 

This section develops a taxonomy of evidence tracing and execution provenance for LLM agents. Unlike prior sections that introduce agent traces and trust requirements, the purpose here is not to summarize individual systems, but to identify the main dimensions along which provenance-aware agent systems can be described and compared. We organize the taxonomy around five dimensions: trace-generating sources, recordable evidence and execution units, provenance relations, recording policy, and trust functions, as summarized in Figure 2. The taxonomy is motivated by two complementary traditions. In provenance modeling, W3C PROV-DM represents provenance through entities, activities, and agents, together with relations such as usage, generation, and deriva-tion World Wide Web Consortium [2013]. This provides a foundation for modeling how data objects or claims are produced. In distributed observability, OpenTelemetry models executions as traces composed of spans and causal relationships, providing a useful analogy for representing multi-step execution paths OpenTelemetry [2026]. However, LLM agents introduce additional semantic and procedural artifacts that are not fully captured by traditional provenance or observability alone, including retrieved evidence, generated claims, tool-call rationales, memory items, natural-language observations, and inter-agent messages. Agent-oriented observability frameworks such as AgentOps and AgentTrace further show that LLM-agent systems require structured records of both execution artifacts and associated context Dong et al. [2024], AlSayyad et al. [2026]. Building on these foundations, we define a taxonomy tailored to the provenance structure of LLM agent execution. 

3.1 Trace Sources 

Trace sources refer to the components of an agent system that generate provenance-relevant artifacts. In simple language-generation settings, the main trace source may be the model output itself. In LLM agents, however, traces can originate from reasoning, retrieval, tool use, memory, environment interaction, and multi-agent communication. Each source contributes different types of evidence and execution records, and each introduces different failure modes. 

Reasoning traces include intermediate thoughts, plans, decompositions, reflections, critiques, and self-corrections generated by the agent. Systems such as ReAct and Reflexion show that reasoning traces can influence action selection and future behavior Yao et al. [2023a], Shinn et al. [2023]. From a provenance perspective, reasoning traces are important because they may explain why an action was selected, why a document was considered relevant, or how an intermediate claim contributed to the final answer.

[2024], Yao et al. [2025]. Tool traces are especially important because tool calls can affect external systems. 3.3 Provenance Relations 

Provenance relations define how trace units are connected. In traditional provenance models, relations such as used ,

wasGeneratedBy , and wasDerivedFrom describe how entities, activities, and agents participate in the production of an artifact World Wide Web Consortium [2013]. These relations provide a useful foundation, but LLM agents require additional relation types because their executions combine semantic evidence use, tool-mediated actions, memory updates, environment observations, and multi-agent communication. In this setting, provenance must capture not only which artifact was used or generated, but also whether a piece of evidence supports a claim, whether a tool output contradicts memory, whether an observation invalidates a plan, or whether an error triggers a recovery step. Building on traditional provenance relations, we define seven relation types for LLM-agent provenance: S UPPORT ,DERIVE , D EPEND -ON , C ONTRADICT , I NVALIDATE , T RIGGER , and U PDATE . These relations capture two complemen-tary aspects of agent execution: semantic evidence relationships, such as support and contradiction, and procedural execution relationships, such as dependency, triggering, and state update. We additionally retain U SE and G ENERATE 

as PROV-compatible base relations. Several of these relations are motivated by evidence-based generation and RAG evaluation. S UPPORT links evidence units to claims, actions, or answers that they justify. For example, a retrieved passage may support an intermediate claim, or a tool output may support a final answer. This relation is central to citation generation, factual precision, source checking, and faithfulness evaluation Gao et al. [2023], Min et al. [2023], Es et al. [2024], Saad-Falcon et al. [2024], Ru et al. [2024], Wu et al. [2024b]. C ONTRADICT links units that conflict semantically, such as a memory item that conflicts with a newly retrieved document. This relation is closely related to hallucination and factual inconsistency analysis, where generated claims may be unsupported or contradicted by evidence Niu et al. [2024], Lin et al. [2025]. INVALIDATE captures a stronger form of negative evidence, where a new observation or tool result makes a prior assumption, plan, or claim no longer usable. For example, a failed API response may invalidate the planned next action. Other relations capture procedural and state-changing aspects of agent execution. D ERIVE captures transformation or summarization, such as deriving a memory item from a previous conversation or deriving a claim from a table. This relation is important for memory-augmented agents, where later behavior may depend on compressed or updated representations of past interactions Park et al. [2023], Packer et al. [2023], Zhong et al. [2024], Xu et al. [2026]. DEPEND -ON represents execution dependency, such as an action depending on a tool result or a tool call depending on generated parameters. This relation is especially important for tool-use safety, because unsafe actions can arise when tool arguments depend on untrusted or injected content Debenedetti et al. [2024, 2025], Costa et al. [2025], Cai et al. [2026], Sequeira et al. [2026]. T RIGGER represents causal or procedural activation, such as an observation triggering a diagnostic tool call or an error triggering replanning. U PDATE captures state changes, including memory updates, environment updates, or database modifications, which are central in interactive benchmarks and tool-agent-user workflows Zhou et al. [2024], Yao et al. [2025]. These relations are useful because they turn raw traces into structured provenance. A chronological log may show that a document was retrieved, a tool was called, and an answer was produced. A provenance graph can additionally show that the document supported one claim, the tool output contradicted another, the contradiction invalidated a previous assumption, and the invalidation triggered a recovery step. Observability and debugging systems similarly show that structured traces are more useful than flat logs for inspection, issue localization, and recovery Dong et al. [2024], AlSayyad et al. [2026], Deshpande et al. [2025], Rorseth et al. [2025]. This relational structure is what allows provenance to support verification, attribution, debugging, safety enforcement, audit, and recovery. 3.4 Tracing Granularity and Timing 

Tracing design depends on both granularity and timing. Granularity determines how detailed the recorded provenance is, while timing determines when traces are collected, checked, or used. 

Run-level tracing records the overall input, major steps, and final output of an agent run. It is useful for coarse audit and reproducibility, but often insufficient for claim-level verification or failure localization. Step-level tracing records each reasoning step, retrieval step, tool call, memory operation, or environment action. This level is useful for debugging and process inspection. This is important for safety because tool misuse often arises from unsafe or untrusted parameter values. Agent-Sentry, for example, uses execution provenance to track the sources of values that flow into tool arguments Sequeira et al. [2026]. Claim-level tracing links individual generated claims to supporting, contradicting, or missing evidence. Token- or span-level tracing is the finest level, but it is expensive and may not always be semantically meaningful for agent-level accountability. Timing introduces another design axis. Debugging systems such as TRAIL and LADYBUG illustrate the value of trace-based failure localization after an agent run Deshpande et al. [2025], Rorseth et al. [2025]. Continuous tracing maintains provenance across sessions, memory updates, and long-horizon workflows, which is especially important for persistent agents and multi-agent systems. Different applications require different trade-offs. A low-risk question-answering agent may only need claim-level citation and post-hoc audit. A tool-using financial, medical, or enterprise agent may require runtime provenance checking at the tool-call and parameter level. A long-term personalized agent may require continuous memory provenance to track how past interactions influence future behavior. Therefore, tracing granularity and timing should be selected based on the risk, persistence, and external impact of the agent workflow. 

3.5 Trust Functions 

The purpose of tracing is not merely to store more logs. Traceability and provenance support specific trust functions that improve the reliability, safety, and governability of LLM agents. We summarize six major functions. 

Verification checks whether claims, actions, or outputs are supported by appropriate evidence. In RAG and evidence-based generation, verification asks whether generated claims are faithful to retrieved context or cited sources Gao et al. 12 From Agent Traces to Trust [2023], Es et al. [2024], Saad-Falcon et al. [2024], Ru et al. [2024]. In agent execution, verification also asks whether tool calls, memory use, and environment actions are justified by the available context. 

Attribution identifies which evidence units, tool outputs, memory items, or agent messages contributed to a claim, action, or final answer. Attribution is narrower than full provenance but essential for accountability. Claim-level attribution is especially important because document-level citation may hide unsupported or partially supported claims Min et al. [2023], Wu et al. [2024b], Schreieder et al. [2025]. 

Debugging uses traces to localize failures. A failed agent run may result from wrong retrieval, malformed parameters, tool execution errors, stale memory, invalid reasoning, or inter-agent miscoordination. Trace-based debugging systems such as TRAIL and LADYBUG show how execution traces can support issue localization and targeted repair Deshpande et al. [2025], Rorseth et al. [2025]. This function turns provenance from a passive record into a diagnostic tool. 

Safety enforcement uses provenance to prevent unsafe actions before they occur. Tool-integrated agents may be exposed to indirect prompt injection, untrusted external content, and unsafe tool outputs Ruan et al. [2024], Zhan et al. [2024], Debenedetti et al. [2024]. Information-flow and runtime enforcement methods such as CaMeL, FIDES, NeuroTaint, AgentSpec, and Agent-Sentry show that safe execution requires tracking the origin, transformation, and influence of information throughout the agent workflow Debenedetti et al. [2025], Costa et al. [2025], Cai et al. [2026], Wang et al. [2025a], Sequeira et al. [2026]. 

Audit reconstructs agent executions for inspection, compliance review, and responsibility analysis. Unlike debugging, which localizes failures, audit asks whether execution records are complete, interpretable, durable, and suitable for external review. This view builds on provenance models such as W3C PROV, which represents entities, activities, agents, and their relations [World Wide Web Consortium, 2013]. For LLM systems, audit trails define chronological records for accountability and post-hoc inspection [Ojewale et al., 2026]. For agent systems, observability and auditability frameworks capture cognitive, operational, and contextual traces across executions [Dong et al., 2024, AlSayyad et al., 2026, Nian et al., 2026]. 

Recovery uses provenance to repair failed executions and control unsafe downstream effects. Self-reflective and tool-interactive methods support retry-based repair using feedback from failed attempts or external tools [Shinn et al., 2023, Gou et al., 2024]. Failure-analysis work highlights trajectory-level localization as a prerequisite for repair [Zhu et al., 2025b]. Transactional planning and semantic recoverability mechanisms support compensation, rollback, and safe reuse of states or tool effects [Chang and Geng, 2025, Yang et al., 2026b]. Checkpoint-restore work shows that rollback can itself create semantically unsafe re-executions [Zheng et al., 2026]. 

# 4 Provenance Representation and Evidence Attribution 

The taxonomy in the previous section identifies what should be traced in LLM-agent systems, including trace sources, evidence units, execution units, provenance relations, granularity, timing, and trust functions. This section addresses the complementary question of how these objects and relations can be represented. We view provenance representation as a family of structures that turn raw traces into inspectable, queryable, and auditable records: structured logs record execution events, execution graphs organize dependencies, evidence and claim-support graphs capture grounding relations, and specialized dependency graphs represent tool, memory, and multi-agent influence. These representations provide the bridge between the taxonomy above and the later discussions of tool-use safety, memory provenance, evaluation, and governance. 

4.1 From Structured Logs to Execution Graphs 

A natural starting point for agent provenance is the structured execution log. Instead of storing only free-form transcripts, structured logs record typed events such as user requests, model outputs, retrieval calls, tool invocations, tool outputs, memory operations, environment observations, inter-agent messages, and final responses. W3C PROV-DM provides the general provenance vocabulary of entities, activities, agents, and relations [World Wide Web Consortium, 2013], while PROV-AGENT adapts this view to agentic workflows by modeling prompts, responses, decisions, tool interactions, and downstream workflow context [Souza et al., 2025]. AgentOps and AgentTrace operationalize this idea for LLM agents: the former frames observability as tracking agent artifacts and their relationships across the agent lifecycle, while the latter proposes structured operational, cognitive, and contextual logs for agent execution [Dong et al., 2024, AlSayyad et al., 2026]. Together, these works show that agent traces should capture intermediate artifacts, execution context, tool interactions, and state changes, rather than only input-output pairs. However, structured logs mainly provide chronological observability, whereas provenance-aware analysis requires dependency structure. 4.2 Evidence Graphs and Claim-Support Attribution 

Evidence attribution requires a more semantic representation than execution ordering. The key question is not only whether evidence was retrieved or cited, but whether a generated claim is actually supported by the evidence it depends on. Prior work motivates this view at different levels of granularity. ALCE treats citations as explicit links between generated answers and supporting sources [Gao et al., 2023], while FActScore decomposes long-form generations into atomic facts whose support can be checked against reliable sources [Min et al., 2023]. SourceCheckup further shows that a cited source may be topically relevant without supporting the specific generated claim [Wu et al., 2024b]. PaperTrail makes this structure explicit by decomposing both LLM-generated answers and source documents into claims and evidence units, then mapping supported claims, unsupported claims, and omitted evidence [Martin-Boyle et al., 2026]. This view motivates evidence and claim-support graphs that connect evidence units to the claims, decisions, or actions they support. Such graphs distinguish evidence availability from evidence use, making grounded generation not only a generation problem but also a provenance representation problem. 

4.3 Static Schemas and Runtime Provenance 

A final design dimension is the distinction between static provenance schemas and dynamic runtime provenance. A static schema defines the object and relation types that may appear in provenance records, such as evidence units, tool calls, memory items, claims, actions, and relations including S UPPORT , D EPEND -ON , C ONTRADICT , U PDATE , and INVALIDATE . W3C PROV-DM provides a general foundation for this view through domain-independent concepts such as entities, activities, agents, and derivation relations [World Wide Web Consortium, 2013]. For LLM agents, this schema must be specialized to include agent-specific objects such as retrieved chunks, tool arguments, memory writes, inter-agent messages, generated claims, and external state changes. Dynamic runtime provenance records the concrete objects and dependencies produced in a particular execution. AgentOps and AgentTrace emphasize runtime trace capture and observability, while Agent-Sentry, FIDES, and NeuroTaint show that runtime provenance can support enforcement, information-flow control, and source-to-sink influence analysis [Dong et al., 2024, AlSayyad et al., 2026, Sequeira et al., 2026, Costa et al., 2025, Cai et al., 2026]. The key trade-off is granularity: fine-grained provenance supports verification, debugging, recovery, and audit, but increases storage overhead, logging complexity, privacy exposure, and annotation burden; coarse-grained provenance is easier to collect, but may fail to identify which evidence supported a claim, which memory item caused an error, or which tool argument crossed a trust boundary. # 5 Execution Provenance in Tool-Using Agents 

Tool use is a central reason why execution provenance is necessary for LLM agents. Once an agent can call external tools, APIs, databases, code interpreters, browsers, or enterprise systems, its behavior is no longer limited to generating text: tool call may retrieve private data, modify external states, send messages, execute code, or trigger irreversible downstream actions. Therefore, trustworthy tool use requires tracing not only final outputs, but also why a tool was called, where its arguments came from, whether its output was reliable, and how the result influenced later decisions. This section reviews provenance mechanisms for tool-call and argument tracing, indirect prompt injection, information-flow control, access boundaries, runtime guardrails, and trace-based debugging. 

5.1 Tool-Call, Argument, and Output Provenance 

Tool-call provenance records the execution path from tool selection to downstream influence: which tool was selected, which arguments were passed, what output was returned, and how that output affected later reasoning or actions. Tool-augmented systems such as Toolformer and ToolLLM show that external tools can expand LLM capabilities by enabling API calls, calculation, search, and other external operations Schick et al. [2023], Qin et al. [2024]. From a provenance perspective, however, the key question is not only whether a tool call succeeds, but whether the call is justified, whether its arguments come from trusted sources, and whether its output is used appropriately. A tool call contains several provenance-critical objects: the selected tool, its schema, argument values, execution result, and downstream consumers. Among these, argument provenance is especially important because unsafe behavior often arises from passing untrusted or incorrectly generated values into otherwise legitimate tools. For example, a calendar or email tool may be safe in general, but unsafe if recipient fields, event descriptions, or message contents are copied from an untrusted webpage containing hidden instructions. Agent-Sentry directly addresses this problem by tracking the sources of values that flow into tool arguments, showing why tool-use safety requires parameter-level provenance rather than only tool-level permission checks Sequeira et al. [2026]. Tool outputs also become provenance-bearing execution units. A returned value may support a claim, update memory, trigger another tool call, modify external state, or conflict with other evidence. In web and tool-agent-user benchmarks, successful execution often depends on interpreting tool outputs correctly, satisfying policy constraints, and applying state-changing actions in the right order Zhou et al. [2024], Yao et al. [2025]. Thus, tool-use provenance should connect upstream sources, tool arguments, tool outputs, and downstream actions into a single dependency chain, rather than merely logging that a tool was called. 

5.2 Indirect Prompt Injection and Tool-Use Contamination 

Tool-using agents are vulnerable to indirect prompt injection because they consume external content that may contain adversarial instructions. Unlike direct prompt injection, where the malicious instruction is placed in the user prompt, indirect injection embeds the instruction in webpages, documents, emails, database entries, tool outputs, or third-party API responses. This attack exploits a core weakness of LLM-integrated systems: the boundary between data and instructions is often ambiguous, so untrusted content may be interpreted as executable guidance rather than passive evidence Greshake et al. [2023], Liu et al. [2023]. Recent benchmarks show that this risk becomes more severe when agents can use tools. InjecAgent evaluates indirect prompt injection in tool-integrated agents, while AgentDojo provides realistic tasks involving tools, private data, and adversarial instructions Zhan et al. [2024], Debenedetti et al. [2024]. ToolEmu further studies unsafe tool execution in an LM-emulated sandbox, allowing dangerous tool-use failures to be evaluated without real-world side effects Ruan et al. [2024]. Together, these works show that malicious external content can cause agents to call unauthorized tools, leak private information, ignore user intent, or perform unintended state-changing actions. From a provenance perspective, indirect prompt injection is an information-flow and influence-tracking problem. The key question is not only whether malicious content was observed, but whether it influenced a later tool call, argument value, memory update, or external action. A flat log may show that a webpage was retrieved and an email tool was later called, but it may not reveal whether the email recipient or body was derived from the webpage, the user instruction, or the model’s own reasoning. Provenance relations such as D ERIVE , D EPEND -ON , T RIGGER , and U SE make this influence explicit. Thus, tool-use provenance should distinguish trusted from untrusted sources and prevent untrusted content from crossing into privileged execution channels. 15 From Agent Traces to Trust 

5.3 Information Flow, Taint Tracking, and Provenance Enforcement 

Recent defenses increasingly frame tool-use safety as an information-flow problem. Rather than relying only on prompting or output filtering, they track where information originates, how it propagates through the agent, and whether it is allowed to influence sensitive actions. This view extends classical information-flow control, where security policies regulate flows among subjects, objects, and security classes Denning [1976], Sabelfeld and Myers [2003]. In LLM agents, the analogous problem is to prevent low-integrity or untrusted external data from controlling high-impact tool calls, memory updates, or state-changing actions. A central challenge is that LLMs may treat untrusted data as executable instruction. CaMeL addresses this by separating control flow from data flow and restricting how external data can influence agent behavior Debenedetti et al. [2025]. Instruction hierarchy and StruQ pursue a related principle by distinguishing privileged instructions from lower-priority user or third-party data through training or structured prompt/data channels Wallace et al. [2024], Chen et al. [2025]. FIDES further formalizes this idea as agent-level information-flow control, tracking confidentiality and integrity labels and enforcing policies during execution Costa et al. [2025]. Semantic taint tracking and execution provenance provide finer-grained enforcement over values, arguments, and dependencies. NeuroTaint propagates taint across neural and symbolic components, while Agent-Sentry tracks whether sensitive tool arguments are influenced by untrusted sources Cai et al. [2026], Sequeira et al. [2026]. These systems show that unsafe behavior often arises from influence , not merely from content: a webpage may be safe to summarize, but unsafe if its hidden instruction becomes an email recipient, database parameter, or authorization signal. Provenance-aware enforcement makes this distinction explicit by connecting sources, transformations, and actions. 

5.4 Access Control and Execution Boundaries 

Information-flow tracking must be paired with access control and execution boundaries. Even if an agent can reason about many tools, it should not be allowed to invoke all tools under all contexts. This follows the classical security principles of least privilege and complete mediation: sensitive actions should require explicit authorization, and access decisions should be checked at the point of use Saltzer and Schroeder [1975]. In tool-using agents, these principles translate into tool permissions, argument constraints, user confirmations, sandboxing, and policy-mediated execution. Recent agent-safety systems operationalize these ideas at runtime. AgentSpec specifies constraints over agent actions and tool use through runtime enforcement rules, while AgentBound constrains execution to reduce unsafe action space Wang et al. [2025a], Bühler et al. [2025]. ToolEmu studies dangerous tool execution in an emulated sandbox, and AgentDojo evaluates tool-use agents under realistic tasks involving tools, private data, and prompt-injection attacks Ruan et al. [2024], Debenedetti et al. [2024]. These systems show that tool-use safety requires more than detecting harmful text: it requires controlling which tools can be used, under what context, with which arguments, and with what external effects. Provenance makes access control context-sensitive. A read-only search tool may safely consume untrusted web content, but an email, database-update, or code-execution tool may require arguments to be derived from trusted user intent, verified tool outputs, or explicit approval. Agent-Sentry illustrates this connection by checking whether sensitive tool arguments are influenced by untrusted sources Sequeira et al. [2026]. Thus, access control is not separate from provenance; it uses provenance to decide whether a proposed action should be allowed, downgraded, sandboxed, confirmed, or blocked. 

5.5 Runtime Guardrails and Pre-/Post-Execution Verification 

Runtime guardrails use provenance information to check agent behavior before, during, or after execution. Pre-execution guardrails decide whether a proposed tool call should be allowed; runtime guardrails monitor information flow and state changes during execution; post-execution verification inspects traces, outputs, and side effects for debugging, audit, and recovery. 5.6 Summary 

Tool-using agents make execution provenance necessary because tool calls can consume untrusted information, expose private data, modify external state, and trigger downstream consequences. The core provenance questions are where tool arguments come from, whether tool outputs are trustworthy, how external content influences actions, and whether execution remains within authorized boundaries. Recent work on prompt-injection benchmarks, information-flow control, taint tracking, execution bounding, runtime guardrails, observability, and trace-based debugging shows a clear shift from output-only safety toward provenance-aware execution control. This shift is central to trustworthy LLM agents: safe tool use requires not only deciding what the agent says, but also tracing and governing what the agent does.

Memory enables LLM agents to maintain information across turns, tasks, sessions, and environments. It allows agents to recall prior observations, user preferences, feedback, tool results, and past decisions, thereby supporting long-term interaction and adaptive behavior. 6.1 Memory Writes, Source Attribution, and Lineage 

Memory provenance begins when a memory item is written. A memory write may originate from user input, retrieved documents, environment observations, tool outputs, feedback, reflections, or messages from other agents. Existing agent memory systems illustrate the centrality of such write operations. Generative Agents store observations in 18 From Agent Traces to Trust a memory stream and use them for reflection and planning [Park et al., 2023]. Reflexion writes verbal feedback into episodic memory to improve future behavior after failures [Shinn et al., 2023]. MemoryBank maintains long-term memory through continuous updates from user interactions [Zhong et al., 2024]. MemGPT frames memory management as virtual context control, where information is moved between memory tiers through explicit read and write operations [Packer et al., 2023]. A-MEM further organizes memory dynamically by linking new memory items with existing ones in an interconnected structure [Xu et al., 2026]. From a provenance perspective, each memory write should carry source and lineage metadata, including its source type, timestamp, authoring agent, supporting evidence, transformation operation, confidence, and update history. This follows the broader provenance principle that generated artifacts should be linked to the entities, activities, and agents that produced them [World Wide Web Consortium, 2013, Souza et al., 2025], and is consistent with agent tracing frameworks that record execution artifacts and their operational context [AlSayyad et al., 2026, Sequeira et al., 2026]. Such lineage is especially important because memory writes often transform information: a document may be summarized into a memory item, multiple observations may be merged into a user profile, or a failed action may be abstracted into a reflection [Park et al., 2023, Shinn et al., 2023, Xu et al., 2026]. Without explicit lineage, these derived memories may later be reused as reliable context even when their origins, assumptions, or supporting evidence are uncertain. Memory provenance should therefore distinguish raw observations, extracted facts, inferred memories, and revised memories. 

6.2 Memory Retrieval, Temporal Validity, and Downstream Influence 

Memory retrieval determines which stored information enters the current execution context. Retrieved memories may influence task decomposition, answer generation, tool selection, argument construction, user modeling, and future memory updates. Long-term memory systems commonly retrieve prior memories to support future interactions: MemoryBank retrieves past memories for personalization [Zhong et al., 2024], MemGPT manages retrieval across memory tiers [Packer et al., 2023], LongMem studies long-term memory mechanisms for language models [Wang et al., 2023b], Mem0 focuses on extracting and retrieving salient information across multi-session dialogues [Chhikara et al., 2025], and A-MEM improves retrieval through dynamic memory linking [Xu et al., 2026]. These systems show that memory retrieval is central to long-horizon agent behavior. However, retrieval also introduces provenance risks. A retrieved memory may be partially relevant, outdated, overly general, derived from a previous hallucination, poisoned by adversarial interaction, or no longer valid in the current environment. Such risks are especially important because memory poisoning and injection attacks show that persistent memory can influence later responses and actions across sessions [Chen et al., 2024, Dong et al., 2025, Tian et al., 2026, Pulipaka et al., 2026]. Temporal validity is therefore a key part of memory provenance: a memory item should record not only its content, but also when it was created or updated, what evidence supported it, whether that evidence remains valid, and whether later observations have contradicted or superseded it. Future provenance-aware retrieval should track not only which memories are retrieved, but also why they are re-trieved and how they are used. Existing memory systems motivate tracking retrieved memory items and retrieval contexts [Packer et al., 2023, Chhikara et al., 2025, Xu et al., 2026], while provenance and observability frameworks motivate recording execution artifacts, sources, and downstream dependencies [World Wide Web Consortium, 2013, Dong et al., 2024, AlSayyad et al., 2026, Souza et al., 2025]. A provenance-aware memory trace should therefore record the triggering query or context, the retrieved memory items, their original sources, relevance signals, validity status, and downstream links to claims, tool calls, actions, final answers, or later memory updates. Such tracing would allow systems to determine whether a final output was influenced by a particular memory and whether that influence was appropriate. It would also support selective invalidation: when a memory is found to be stale, unsupported, private, or contaminated, the system could identify affected claims, actions, and derived memories rather than treating the execution history as opaque [Wang et al., 2025b, Wei et al., 2025, Sequeira et al., 2026]. 

6.3 Conflicting, Contradictory, and Poisoned Memory 

Memory provenance must also account for conflict, contradiction, and contamination: stored memories may support incompatible claims, become invalidated by later evidence, or encode malicious content that is later reused as trustworthy context. Recent security work makes this risk explicit. AgentPoison shows that poisoned long-term memory or RAG knowledge bases can serve as backdoor substrates for LLM agents [Chen et al., 2024]. Memory-injection attacks further demonstrate that malicious records can be induced through ordinary interactions without direct access to the memory store [Dong et al., 2025]. Environment-injected and sleeper memory poisoning studies show that untrusted observations from webpages, documents, or repositories can silently enter memory, persist across sessions, and later steer agent behavior [Zou et al., 2026, Pulipaka et al., 2026]. These findings suggest that memory quality cannot be evaluated 19 From Agent Traces to Trust only by recall or personalization performance; it also depends on source trust, write-path legitimacy, conflict status, contamination risk, cross-session persistence, and downstream activation. This problem also connects memory provenance to tool-use safety and information-flow control. Fides tracks confi-dentiality and integrity labels to control unsafe information flows in AI agents [Costa et al., 2025], while NeuroTaint emphasizes that unsafe influence can persist through semantic transformations and memory rather than appearing only as exact text reuse [Cai et al., 2026]. Indirect prompt-injection benchmarks such as InjecAgent and AgentDojo further show that untrusted external content can manipulate tool-using agents [Zhan et al., 2024, Debenedetti et al., 2024]. When such content is written into persistent memory, the risk extends beyond a single execution and becomes a long-term provenance problem. 

6.4 Memory-Grounded Verification and Long-Term Audit 

Memory-grounded verification extends evidence tracing from retrieved documents and current tool outputs to persistent memory. When a claim or tool action relies on memory, the system should link that memory item to the observation, document, tool call, user statement, or reflection that created it, as well as to later revisions, conflicts, invalidations, and downstream uses. Claim-evidence interfaces such as PaperTrail motivate this granularity by decomposing LLM answers and source documents into discrete claims and evidence units [Martin-Boyle et al., 2026], while agent observability systems such as AgentOps and AgentTrace provide foundations for recording agent executions and structured operational, cognitive, and contextual logs [Dong et al., 2024, AlSayyad et al., 2026]. However, memory audit requires more than recording memory operations inside a single run. It requires persistent lineage across memory writes, retrieval events, updates, conflicts, invalidations, and later uses. Provenance-based runtime systems such as Agent-Sentry are especially relevant because they use execution provenance graphs to record the origins of tool-call argument values and support source-aware enforcement [Sequeira et al., 2026]. Long-term memory requires an analogous structure for persistent evidence. Despite progress in long-term memory architectures and systems such as MemGPT, A-MEM, and Mem0 [Packer et al., 2023, Xu et al., 2026, Chhikara et al., 2025], memory provenance remains underdeveloped. Existing systems often optimize for recall, personalization, efficiency, or long-term coherence, but provide limited support for tracing memory origins, temporal validity, conflicts, contamination, privacy exposure, and downstream influence. This gap is critical because persistent memory itself can become an attack surface. AgentPoison shows that long-term memory or RAG knowledge bases can be poisoned to induce malicious downstream behavior [Chen et al., 2024]; InjecMEM shows that targeted memory injection can steer later responses through ordinary agent interactions [Tian et al., 2026]; and sleeper memory poisoning demonstrates that adversarial external context can implant fabricated memories that remain dormant and influence future conversations or actions [Pulipaka et al., 2026]. Memory privacy studies further show that sensitive information stored in agent memory can be exposed through adversarial interactions [Wang et al., 2025b]. Defense work such as A-MemGuard suggests that agent memory should support self-checking and self-correction rather than only input-time filtering [Wei et al., 2025]. Future memory systems therefore need provenance-aware mechanisms for source attribution, versioning, conflict detection, taint tracking, privacy-aware access control, stale-memory invalidation, and evidence-grounded verification, shifting memory from a capability mechanism to an auditable basis for accountable agent behavior.

RAG and attribution benchmarks provide the closest foundation for evidence tracing because they evaluate whether generated outputs are grounded in retrieved or cited evidence [Gao et al., 2023, Es et al., 2024, Saad-Falcon et al., 2024, Min et al., 2023, Niu et al., 2024, Ru et al., 2024, Wu et al., 2024b]. These benchmarks generally focus on the relationship between a final answer and its supporting sources, rather than the full execution trajectory of an agent. Nevertheless, they provide important metrics and datasets for evaluating evidence recall, citation quality, faithfulness, and claim-level support. ALCE evaluates whether LLMs can generate long-form answers with citations and whether those citations support the corresponding generated content [Gao et al., 2023]. It is particularly relevant to evidence tracing because it treats citations as explicit links between answer spans and external sources. RAGAS provides reference-free evaluation for RAG systems, including metrics such as faithfulness, context relevance, and answer relevance [Es et al., 2024]. ARES similarly evaluates RAG systems through context relevance, answer faithfulness, and answer relevance, using synthetic data and lightweight model judges [Saad-Falcon et al., 2024]. These frameworks operationalize the idea that an answer should be supported by retrieved evidence, although they mainly target answer-level or context-level evaluation. More fine-grained benchmarks move closer to claim-level provenance. RAGChecker diagnoses RAG systems by separating retrieval and generation errors, making it useful for component-level and module-level analysis [Ru et al., 2024]. RAGTruth provides word-level hallucination annotations for retrieval-augmented generation, allowing evaluation of unsupported and contradicted content at a finer granularity [Niu et al., 2024]. FActScore decomposes long-form generation into atomic facts and evaluates whether each fact is supported by reliable sources [Min et al., 2023]. SourceCheckup evaluates whether LLM-cited medical references are relevant and supportive, highlighting that citation presence does not necessarily imply evidence support [Wu et al., 2024b]. Classic fact-verification benchmarks such as FEVER also provide foundations for claim-evidence evaluation by classifying claims as supported, refuted, or not verifiable [Thorne et al., 2018]. These benchmarks are important for evidence tracing, but they remain limited from an agent-provenance perspective. Their primary evaluation targets are answer-source grounding, citation support, faithfulness, hallucination annotation, or claim-level factual support [Gao et al., 2023, Es et al., 2024, Saad-Falcon et al., 2024, Min et al., 2023, Niu et al., 2024, Ru et al., 2024, Wu et al., 2024b]. 7.2 Benchmarks for Agent Execution Traces and Tool-Use Safety 

Agent benchmarks extend evaluation from static answer generation to interactive, multi-step execution. They are useful for provenance research because they expose trajectories, tool calls, environment states, and intermediate decisions. AgentBench evaluates LLMs as agents across multiple environments, emphasizing reasoning and decision-making in interactive settings [Liu et al., 2024]. WebArena provides a realistic web environment for autonomous agents, where agents must perform long-horizon tasks through browser actions and environmental feedback [Zhou et al., 2024]. ToolBench and ToolLLM evaluate tool-using agents through API selection, argument generation, and tool execution paths [Qin et al., 2024]. τ -bench focuses on tool-agent-user interaction in real-world domains, evaluating not only final responses but also policy following, tool use, user interaction, and final database state [Yao et al., 2025]. Trace-oriented debugging benchmarks provide a more direct connection to execution provenance. TRAIL introduces trace reasoning and agentic issue localization, using human-annotated traces to evaluate whether systems can identify where agent failures originate [Deshpande et al., 2025]. This is important because provenance is not only about recording execution, but also about using traces to localize errors. AgentTrace provides a structured logging framework for agent observability, capturing operational, cognitive, and contextual logs that can support downstream debugging and audit [AlSayyad et al., 2026]. Tool-use safety benchmarks evaluate whether agents remain safe when tools introduce external risks. ToolEmu uses an LM-emulated sandbox to evaluate high-risk tool-use behavior, including scenarios where agents may cause privacy leakage, financial loss, or other harmful external effects [Ruan et al., 2024]. InjecAgent benchmarks indirect prompt injection in tool-integrated LLM agents, showing how malicious external content can influence tool-using behavior [Zhan et al., 2024]. AgentDojo provides a dynamic environment for evaluating prompt injection attacks and defenses in realistic tool-use settings [Debenedetti et al., 2024]. These benchmarks are especially relevant to provenance because unsafe behavior often arises from hidden information flow from untrusted sources into tool arguments or actions. Recent work evaluates more explicit execution-provenance and information-flow mechanisms. Agent-Sentry introduces execution provenance graphs to bound LLM-agent behavior and detect out-of-bounds tool calls [Sequeira et al., 2026]. NeuroTaint and TaintBench evaluate semantic taint and source-to-sink influence in LLM agents, emphasizing that unsafe influence may be transformed through summarization, inference, or memory rather than copied verbatim [Cai 22 From Agent Traces to Trust et al., 2026]. Fides applies information-flow control to secure AI agents by tracking confidentiality and integrity labels [Costa et al., 2025]. AgentSpec evaluates runtime policy enforcement through customizable constraints [Wang et al., 2025a]. AgentBound extends this line of work to execution boundaries and access control in MCP-style agent-tool ecosystems [Bühler et al., 2025]. Together, these benchmarks and systems shift evaluation from “can the agent use tools?” to “can the agent use tools safely, with traceable and enforceable provenance?” 

7.3 Memory and Multi-Agent Provenance Evaluation 

Memory and multi-agent settings introduce additional provenance challenges because information may persist across sessions, be updated over time, or propagate across agents. Existing memory benchmarks and systems mainly evaluate recall, personalization, long-term consistency, memory updating, retrieval, or multi-session task performance [Zhong et al., 2024, Packer et al., 2023, Xu et al., 2026, Chhikara et al., 2025, Maharana et al., 2024, Kim et al., 2026]. MemoryBank evaluates long-term memory recall, continuous updating, and personalization [Zhong et al., 2024]; MemGPT makes memory read and write operations explicit through hierarchical memory management [Packer et al., 2023]; A-MEM organizes agent memory through dynamically evolving interconnected memory structures [Xu et al., 2026]; Mem0 evaluates memory extraction, consolidation, and retrieval in multi-session dialogue settings [Chhikara et al., 2025]; LOCOMO evaluates long-term conversational memory [Maharana et al., 2024]; and MemoryArena evaluates memory use in interdependent multi-session agentic tasks [Kim et al., 2026]. Existing memory and graph-memory surveys further summarize progress in memory mechanisms, evaluation protocols, and graph-based memory systems [Zhang et al., 2025, Du, 2026, Yang et al., 2026a]. From a provenance perspective, these benchmarks provide useful but incomplete proxies for memory provenance. They often test whether an agent remembers useful information, but rarely evaluate whether a memory item is traceable to its source, temporally valid, consistent with newer evidence, protected from contamination, or responsible for a downstream action. Dedicated memory-provenance evaluation should therefore assess memory source attribution, update history, conflict resolution, stale-memory invalidation, contamination detection, and downstream influence, which remain underdeveloped in current memory evaluation protocols [Maharana et al., 2024, Kim et al., 2026, Zhang et al., 2025, Du, 2026]. Multi-agent systems introduce another level of provenance complexity because evidence, assumptions, messages, and errors may propagate across agent boundaries. AutoGen and CAMEL provide representative frameworks for multi-agent communication, role assignment, and collaborative task solving [Wu et al., 2024a, Li et al., 2023]. However, evaluating multi-agent provenance requires more than measuring final task success: it requires identifying which agent contributed which evidence, which communication step introduced an error, and whether responsibility can be attributed across agents. MAST analyzes multi-agent failures related to specification, inter-agent misalignment, and task verification [Cemri et al., 2026], while TRAIL evaluates issue localization from agent traces [Deshpande et al., 2025]. These works suggest that future multi-agent provenance benchmarks should evaluate evidence propagation, communication-level error introduction, cross-agent dependency, and responsibility attribution, rather than only final coordination success.

7.5 Limitations of Current Evaluation Protocols 

Existing benchmarks provide useful foundations for evaluating retrieval grounding, tool use, memory, web interaction, and multi-agent behavior, but they remain insufficient for full evidence tracing and execution provenance. RAG and attribution benchmarks mainly focus on answer-source grounding or citation quality [Gao et al., 2023, Ru et al., 2024, Wu et al., 2024b]; tool-use and agent benchmarks emphasize task success, unsafe tool behavior, prompt injection, or environment interaction [Liu et al., 2024, Zhou et al., 2024, Ruan et al., 2024, Debenedetti et al., 2024, Yao et al., 2025]; and memory or multi-agent benchmarks evaluate recall, personalization, coordination, or failure modes [Maharana et al., 2024, Kim et al., 2026, Deshpande et al., 2025, Cemri et al., 2026]. These benchmarks therefore cover important components, but rarely evaluate the complete provenance chain from evidence and tool outputs to memory updates, intermediate claims, final answers, and downstream actions. A key limitation is the lack of unified execution traces and provenance-relation annotations. Trace-oriented benchmarks and observability frameworks have made useful progress, but they still differ in what artifacts they record and how provenance is represented [Dong et al., 2024, AlSayyad et al., 2026, Souza et al., 2025, Deshpande et al., 2025, Cemri et al., 2026]. Few benchmarks jointly capture user requests, retrieved evidence, tool calls and outputs, memory reads and writes, inter-agent messages, intermediate claims, final responses, and state changes. Even fewer annotate relations such as S UPPORT , C ONTRADICT , D EPEND -ON , U PDATE , and I NVALIDATE , despite the availability of general provenance concepts in standards such as W3C PROV-DM [World Wide Web Consortium, 2013]. As a result, current evaluation often measures correctness, faithfulness, or attack success indirectly, rather than directly assessing trace completeness, provenance accuracy, claim support, memory validity, or cross-agent responsibility. Recovery and governance are also under-evaluated. # 8 Open Problems and Future Directions 

Despite rapid progress in RAG attribution, agent observability, tool-use safety, memory systems, graph-based repre-sentation, debugging, and multi-agent evaluation, evidence tracing and execution provenance for LLM agents remain fragmented. Existing systems can record execution artifacts, but logging alone does not explain how evidence, tools, memory, environment observations, and inter-agent messages shape an agent’s final answer or action. This section summarizes open problems toward provenance-aware agents that support verification, auditing, attribution, safety, recovery, and governance. 

8.1 Toward Unified and Interoperable Trace Schemas 

A major challenge is the lack of unified trace schemas for LLM agents. Existing agent frameworks, observability tools, and benchmarks record different artifacts using different formats. Some systems log prompts and model responses; others log tool calls, retrieved documents, memory reads and writes, environment observations, execution metadata, or error events [Dong et al., 2024, AlSayyad et al., 2026, Souza et al., 2025, Deshpande et al., 2025, Cemri et al., 2026]. 25 From Agent Traces to Trust Few provide interoperable schemas that jointly represent reasoning steps, retrieval, tool calls, tool outputs, memory operations, inter-agent messages, intermediate claims, final answers, and external state changes. Traditional provenance and tracing standards provide useful starting points, although they were not designed specifically for LLM agents. W3C PROV-DM defines general provenance concepts such as entities, activities, agents, usage, genera-tion, and derivation [World Wide Web Consortium, 2013]. OpenTelemetry and OpenLineage provide infrastructure-level abstractions for distributed traces and data workflow lineage [OpenTelemetry, 2026, OpenLineage Contributors, 2026]. Agent-specific frameworks such as AgentOps and AgentTrace move closer to LLM-agent observability by recording agent artifacts and structured operational, cognitive, and contextual logs [Dong et al., 2024, AlSayyad et al., 2026]. PROV-AGENT further adapts provenance modeling to agentic workflows [Souza et al., 2025]. However, a unified schema is still needed to connect operational execution events with semantic evidence relations across retrieval, tool use, memory, multi-agent communication, and state changes. Future LLM-agent provenance schemas should connect operational execution events with semantic evidence relations. Existing standards and frameworks provide useful building blocks [World Wide Web Consortium, 2013, OpenTelemetry, 2026, OpenLineage Contributors, 2026, Souza et al., 2025], but they do not yet fully cover provenance across retrieval, tool use, memory, multi-agent communication, and external state changes. A unified schema should represent key entities, such as claims, evidence units, tool observations, memory items, environment states, and agent messages, together with relations such as S UPPORT , C ONTRADICT , D EPEND -ON , U PDATE , and I NVALIDATE . Such schemas would make agent traces easier to compare, replay, annotate, audit, and exchange across frameworks. 

8.2 From Coarse Attribution to Claim-Level and Semantic Provenance 

Current attribution methods often operate at coarse granularity, such as answer-level citation, context-level faithfulness, or step-level logging [Gao et al., 2023, Ru et al., 2024, Wu et al., 2024b, Schreieder et al., 2025]. However, LLM-agent outputs frequently contain multiple claims, intermediate assumptions, and action-relevant inferences. A final response may be partially correct: one claim may be supported by retrieved evidence, another may be inferred beyond the source, and another may be contradicted by a tool output or memory item. This motivates claim-level provenance, which builds on fine-grained factuality and attribution evaluation but extends it to agentic execution settings [Min et al., 2023, Niu et al., 2024]. Existing work provides useful foundations. ALCE studies citation generation and citation quality [Gao et al., 2023]; FActScore evaluates atomic fact support [Min et al., 2023]; RAGTruth provides fine-grained hallucination annota-tions [Niu et al., 2024]; RAGChecker diagnoses RAG systems at a finer granularity [Ru et al., 2024]; and SourceCheckup evaluates whether cited sources are actually relevant and supportive [Wu et al., 2024b]. Broader surveys of attribution, citation, and quotation further show that evidence-based text generation still lacks unified definitions and evaluation protocols [Schreieder et al., 2025]. For agents, the problem is harder because provenance must track semantic influence across execution. LLMs may paraphrase, summarize, combine, infer, and transform information before using it in a claim or action. Evidence may pass through retrieval, tool calls, memory updates, reflection, or another agent’s intermediate judgment. String-level matching is therefore insufficient. NeuroTaint highlights this issue by emphasizing semantic taint and causal influence in LLM agents [Cai et al., 2026]. Future work should develop methods for tracing semantic dependencies, including cases where evidence is transformed, compressed, aggregated, or indirectly used across multiple steps. 

8.3 Memory and Multi-Agent Provenance 

Memory provenance remains underexplored. Existing memory systems allow agents to store, retrieve, update, and reuse information across sessions, but they often provide limited support for tracing where memory items came from, whether they remain valid, whether they conflict with newer evidence, and how they influence later claims or actions. Memory surveys and graph-memory surveys have reviewed memory architectures, evaluation methods, and relational memory structures [Zhang et al., 2025, Du, 2026, Yang et al., 2026a], but memory as provenance-bearing evidence remains a distinct open problem. Future memory systems should record source attribution, write time, update history, retrieval context, downstream usage, conflict status, and invalidation events. Memory should not be treated as an unqualified context reservoir. Instead, memory items should carry provenance metadata indicating whether they were derived from user input, retrieved evidence, tool outputs, environment observations, reflections, or messages from other agents. This is especially important when memory affects tool calls, personalization, or long-term decision-making. Security work such as AgentPoison and memory-injection attacks shows that poisoned or maliciously induced memory can influence future agent behavior without modifying model weights [Chen et al., 2024, Dong et al., 2025]. 26 From Agent Traces to Trust Multi-agent provenance introduces additional complexity. Frameworks such as AutoGen and CAMEL demonstrate that agents may communicate, delegate tasks, and collaboratively solve problems [Wu et al., 2024a, Li et al., 2023]. MAST shows that multi-agent systems can fail because of specification errors, inter-agent misalignment, weak verification, and coordination failures [Cemri et al., 2026]. In such settings, evidence and errors can propagate across agents. Future provenance models should identify which agent introduced a claim, which agent verified it, which agent reused it, which message carried it, and where responsibility lies when a multi-agent system fails. This requires provenance representations for delegation, role assignment, message-level influence, cross-agent verification, and failure propagation. 

8.4 Provenance-Aware Runtime Safety and Recovery 

Runtime safety for LLM agents requires more than output filtering. In tool-using agents, dangerous behavior may arise when untrusted content influences tool arguments, when private information flows into public outputs, when previously stored context affects a privileged action, or when an agent invokes a tool outside its intended boundary. Guardrails that inspect only the final answer or the current tool call may miss these risks. Provenance-aware guardrails should instead reason over the source chain behind an action, including which evidence, memory items, tool outputs, and external observations influenced the action [Zhan et al., 2024, Debenedetti et al., 2024, 2025, Costa et al., 2025, Sequeira et al., 2026, Bühler et al., 2025]. Recent work points toward this direction. ToolEmu evaluates high-risk tool-use behavior [Ruan et al., 2024]; InjecAgent and AgentDojo evaluate indirect prompt injection in tool-integrated agents [Zhan et al., 2024, Debenedetti et al., 2024]; CaMeL separates trusted control flow from untrusted data flow [Debenedetti et al., 2025]; Fides applies information-flow control to AI agents [Costa et al., 2025]; NeuroTaint traces semantic influence [Cai et al., 2026]; AgentSpec provides runtime constraint enforcement [Wang et al., 2025a]; Agent-Sentry uses execution provenance graphs to bound agent behavior [Sequeira et al., 2026]; and AgentBound studies execution boundaries and access control in MCP-style agent-tool ecosystems [Bühler et al., 2025]. However, runtime provenance remains difficult. Guardrails must balance safety and utility, avoid excessive false positives, and operate under incomplete or noisy traces. Moreover, most current systems focus on detection or blocking, while recovery remains underdeveloped. Future provenance-aware agents should not only detect unsafe or unsupported execution, but also repair it. For example, an agent should be able to invalidate stale memory, quarantine contaminated evidence, retry a failed tool call with corrected parameters, ask for human approval, roll back an unsafe state change, or replay execution from a safe checkpoint. This requires provenance representations that support intervention and recovery, not just explanation, building on provenance modeling, runtime enforcement, information-flow control, and execution-provenance mechanisms [World Wide Web Consortium, 2013, Souza et al., 2025, Costa et al., 2025, Wang et al., 2025a, Sequeira et al., 2026]. 

8.5 Realistic Benchmarks, Privacy, and Governance 

Current benchmarks still lack realistic full execution traces. RAG benchmarks provide evidence labels but usually omit tools and memory. Tool-use safety benchmarks provide attacks and tool calls but often lack claim-level evidence labels. Memory benchmarks evaluate recall or personalization but rarely evaluate memory lineage, conflict handling, or downstream influence. Multi-agent benchmarks analyze failures but often lack complete provenance annotations. Benchmarks such as AgentBench, WebArena, τ -bench, TRAIL, MAST, LOCOMO, and MemoryArena provide important foundations [Liu et al., 2024, Zhou et al., 2024, Yao et al., 2025, Deshpande et al., 2025, Cemri et al., 2026, Maharana et al., 2024, Kim et al., 2026], but no existing benchmark fully evaluates evidence tracing and execution provenance across retrieval, tool use, memory, multi-agent communication, safety attacks, and recovery. Future benchmarks should move from isolated answer- or task-level evaluation toward trace-level evaluation. Building on provenance standards and agent observability frameworks [World Wide Web Consortium, 2013, Dong et al., 2024, AlSayyad et al., 2026, Souza et al., 2025], they should capture complete execution records across retrieval, tool use, memory, environment interaction, inter-agent communication, and state changes. They should further annotate provenance relations such as support, contradiction, dependency, triggering, update, and invalidation. Recent trace-oriented benchmarks such as TRAIL and MAST show the value of annotated execution traces for failure analysis [Deshpande et al., 2025, Cemri et al., 2026], but future benchmarks should extend this idea to claim-level evidence support, memory validity, unsafe influence detection, auditability, and recovery. Privacy and governance are equally important. Provenance traces may contain sensitive user data, confidential documents, credentials, tool outputs, API arguments, internal reasoning artifacts, and long-term memory items. Observability frameworks such as AgentOps and AgentTrace provide useful foundations for collecting traces [Dong et al., 2024, AlSayyad et al., 2026], and security mechanisms such as Fides highlight the need to track confidentiality 27 From Agent Traces to Trust and integrity constraints [Costa et al., 2025]. However, future systems must also support trace minimization, access control, anonymization, retention policies, encryption, and selective disclosure. Without such mechanisms, provenance infrastructure itself may become a source of privacy and compliance risk. Overall, realistic provenance-aware benchmarks must be developed together with privacy and governance mechanisms, so that agent systems can be evaluated not only by task success, but also by traceability, auditability, safety, and recoverability. 

# 9 Conclusion 

LLM agents are moving beyond isolated text generation toward systems that retrieve evidence, plan intermediate steps, invoke tools, update memory, interact with environments, and coordinate with other agents. This shift makes final-answer correctness insufficient for trustworthiness. Reliable agent systems require not only accurate outputs, but also the ability to trace where answers come from, which evidence supports them, how tools and intermediate actions contribute to them, and where failures or unsafe influences enter the execution process. In this survey, we position evidence tracing and execution provenance as a unified perspective for studying trustworthy LLM agents. We organize retrieval grounding, citation faithfulness, tool-use logging, memory auditing, agent evaluation, and security risks around a common question: how information, decisions, and actions can be traced across the full agent lifecycle. We review key provenance representations, including structured logs, execution graphs, evidence graphs, tool dependency graphs, memory provenance, and multi-agent communication traces, and discuss their roles in attribution, debugging, verification, safety enforcement, evaluation, and recovery. Our review shows that the field has made important progress, but remains fragmented. Existing systems often record sources or tool calls, yet they rarely provide fine-grained claim-level support, semantic influence tracking, memory lineage, cross-agent provenance propagation, or robust tracing under adversarial conditions. Looking forward, evidence tracing and execution provenance should become a first-class infrastructure layer for reliable agent systems. We hope this survey helps consolidate this emerging research space and motivates the development of traceable, auditable, and recoverable LLM agents.
