## 摘要
大型语言模型（LLM）智能体在解决现实任务时会积累丰富的情景轨迹，但这些经验能否提炼为可复用的程序性技能仍不明确。我们引入SkillEvolBench，一个用于评估从经验复用到技能形成这一步骤的诊断性基准。它包含跨越六个真实世界智能体环境的180个任务，组织成具有共享潜在程序、以角色为条件的任务家族。智能体从习得任务中学习，利用压缩轨迹和验证器反馈更新外部技能库，然后面对冻结部署任务，测试上下文偏移、对抗性捷径和组合能力。通过对比自我生成和策展启动的技能演化与无技能及原始轨迹对照，SkillEvolBench将程序性抽象与基础能力、策展先验知识及对情景轨迹的直接复用分离开来。在十种模型配置和三种智能体框架下，我们发现当前智能体往往只能局部适应，却很少形成稳健可复用的技能。基于技能的方式可以改善习得或重放，个别模型有时在特定部署维度上有所提升，但这些提升在冻结部署下并不稳定。原始轨迹复用经常优于提炼后的技能，表明当前的抽象程序丢弃了对于未来任务仍有益的上下文和程序性线索。容量和成本分析进一步表明，写入更多技能或更大规模的三级资源库并不足够：额外的更新可能提高覆盖率，却会引入特定于回放片段的漂移和程序性杂乱。这些发现使SkillEvolBench成为一个测试平台，用于度量一次性经验何时转变为持久的程序性知识，而非任务局部记忆。

## Abstract
Large language model (LLM) agents accumulate rich episodic trajectories while solving real-world tasks, but it remains unclear whether such experience can be distilled into reusable procedural skills. We introduce SkillEvolBench, a diagnostic benchmark for evaluating this step from experience reuse to skill formation. It contains 180 tasks across six real-world agent environments, organized into role-conditioned task families with shared latent procedures. Agents learn from acquisition tasks, update an external skill library using compacted trajectories and verifier feedback, and then face frozen deployment tasks testing context shift, adversarial shortcuts, and composition. By comparing self-generated and curated-start skill evolution against no-skill and raw-trajectory controls, SkillEvolBench separates procedural abstraction from base capability, curated prior knowledge, and direct reuse of episodic traces. Across ten model configurations and three agent harnesses, we find that current agents often adapt locally but rarely form robust reusable skills. Skill-based conditions can improve acquisition or replay, and individual models sometimes gain on specific deployment axes, but these gains are unstable under frozen deployment. Raw-trajectory reuse frequently outperforms distilled skills, suggesting that current abstraction procedures discard contextual and procedural cues that remain useful for future tasks. Capacity and cost analyses further show that writing more skills or larger Tier-3 resource libraries is not sufficient: additional updates can improve coverage while introducing episode-specific drift and procedural clutter. These findings position SkillEvolBench as a testbed for measuring when one-off experience becomes durable procedural knowledge rather than task-local memory.

*   **整体含义**：本研究旨在**诊断从“经验复用到技能形成”这一关键步骤的现状和能力瓶颈**。它并非仅仅测试技能是否对当前任务有帮助，而是评估智能体在未见过的、更困难的部署任务上，能否利用先前从经验中演化出的技能来泛化、抵御干扰并组合运用。

### 2. 论文提出的方法论

*   **核心思想：技能演化协议 (Skill Evolution Protocol)**
    整个方法构建了一个结构化的持续学习和评估循环，其核心是让智能体在一个环境（Environment）内进行“终身的”学习与测试。流程如图3所示，包含五个关键阶段：
    1.  **初始化**：为每个环境激活一个空白的技能库。依据实验条件，该库可能以空白（自我生成）、仅有元数据的技能（Zero-Shot）或包含一个“留有缺口的策展技能 (gap-exposed curated skill)”作为起点。
    2.  **习得 (Acquisition)**：智能体在一个任务家族内，按顺序解决`规范 (canonical)`、`强化 (enriched)`和`变体 (variant)`三种角色任务。每次尝试都会生成一个压缩后的轨迹摘要和结构化的验证器反馈。
    3.  **技能更新**：一个独立于任务执行智能体的**技能作者 (Skill Author)** 模块，会依据同一家族的习得历史（压缩轨迹+反馈）和当前技能库状态，决定执行三种操作之一：**编写新技能、修订现有技能、或不作任何操作**。更新策略可以是仅在失败后触发，也可以是每次尝试后都触发（Always）。
    4.  **冻结与部署 (Frozen Deployment)**：技能库被冻结，智能体在**不可更新技能**的条件下，面对三种更具挑战性的部署任务：`上下文偏移 (context shift)`、`对抗性捷径 (adversarial shortcut)` 和 `技能组合 (composition)`。成功唯一取决于先前形成的技能是否足够泛化。
    5.  **重放 (Replay)**：在一个环境任务完成后，用最终冻结的技能库重新运行最初的习得任务，以衡量技能的“局部恢复”能力，并与向更难任务的迁移能力进行区分。之后重置技能库，进入下一个独立环境。

*   **关键技术细节**：
    *   **任务家族设计**：每个任务家族都围绕一个潜在的、可重复的程序进行设计，包含6种角色，以系统地测试技能形成的不同方面。
    *   **差距暴露的策展技能**：提供的初始技能**并非完美**。它包含解决基础任务的核心程序，但有意留下缺口，迫使智能体必须通过经验学习来填补这些缺口，以应对增强、对抗、组合等任务。
    *   **轨迹压缩 (Trajectory Compaction)**：为了不将过于冗长且包含噪声的完整轨迹直接喂给技能作者，系统会对轨迹进行确定性的信息压缩，只保留关键事件（如工具调用、观察结果、错误、验证信息）的摘要。
    *   **结构化验证器反馈**：反馈不仅仅是成功或失败，还包括失败的测试用例、输出违规、过程违规等诊断信息，为技能作者提供具体的修改依据。

### 6. 论文的主要结论与发现

*   **局部适应不等于可复用技能**：当前智能体能在经验基础上进行局部适应（提升LSR或RSR），但这些提升并**不能稳定地迁移到**需要泛化、鲁棒性和组合能力的冻结部署任务上。
*   **存在“有损抽象瓶颈”**：一个核心发现是，**直接复用原始轨迹的效果常常优于复用从这些轨迹中提炼出的抽象技能**。这表明当前的技能抽象过程丢失了大量对后续任务有用的上下文和程序性线索（图4）。
*   **更多技能更新并非总是更好**：采用“总是更新”策略虽然可能增加问题覆盖率，但也可能引入**特定于某次尝试的漂移和程序性杂乱**，反而损害性能，甚至不如仅在失败时更新的策略。
*   **更大的技能库容量并不解决问题**：通过强制生成三级资源（Tier-3，如脚本、参考文件等）来增加技能库的大小和细节，并不能可靠提升性能。关键在于**有选择性地保留稳定的程序，而非简单地存储更多信息**。
*   **性能高度依赖任务结构和模型**：任务环境和部署角色是性能差异的巨大来源，同时，特定模型与技能条件的匹配度也至关重要，没有一种技能演化策略对所有模型都普适有效。
